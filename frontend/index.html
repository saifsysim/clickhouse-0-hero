ClickHouse Table Engines

Every table in ClickHouse uses an engine that defines how data is stored, merged, and queried. Choose the right engine for your workload.

MergeTree

MergeTree

The foundation of all ClickHouse engines. Data is written in parts, then merged in the background. Supports ORDER BY, PARTITION BY, sparse primary indexes, and TTL.

-- Used for: telemetry_events, app_logs CREATE TABLE events ( timestamp DateTime, service LowCardinality(String), user_id String ) ENGINE = MergeTree() PARTITION BY toYYYYMM(timestamp) ORDER BY (service, timestamp) TTL timestamp + INTERVAL 90 DAY;
📡 Telemetry 📋 Logs 📈 Time-series
SummingMergeTree

SummingMergeTree

Automatically sums numeric columns during merges. Perfect for counters and accumulators — no aggregation needed at query time.

-- Used for: cost_usage CREATE TABLE cost_usage ( timestamp DateTime, service LowCardinality(String), team LowCardinality(String), cost_usd Float64, api_calls UInt32 ) ENGINE = SummingMergeTree( (cost_usd, api_calls) -- columns to sum ) ORDER BY (team, service, timestamp);
💰 Cost Tracking 📊 Budget Rollup
AggregatingMergeTree

AggregatingMergeTree

Stores partial aggregate states (e.g. HyperLogLog for uniq). Used with Materialized Views for blazing-fast pre-computed rollups.

-- Backing table for the hourly rollup MV CREATE TABLE telemetry_hourly_agg ( hour DateTime, service LowCardinality(String), event_count AggregateFunction(count), unique_users AggregateFunction(uniq, String) ) ENGINE = AggregatingMergeTree() ORDER BY (service, hour); -- Materialized View feeds it automatically CREATE MATERIALIZED VIEW mv_hourly TO telemetry_hourly_agg AS SELECT toStartOfHour(ts) AS hour, service, countState() AS event_count, uniqState(user_id) AS unique_users FROM events GROUP BY hour, service;
📉 Rollups 🔭 Pre-aggregation
Materialized View

Materialized View — trigger, not a snapshot

In ClickHouse a Materialized View is a trigger that fires on every INSERT — it runs its SELECT on only the new rows (the delta) and writes the result into a target table. Unlike PostgreSQL's REFRESH MATERIALIZED VIEW which re-runs the full query on a schedule, ClickHouse MVs update continuously and incrementally. No cron. No scheduler. Written once, runs forever.

Step 1 — Source table (raw events)
CREATE TABLE page_views ( user_id String, domain String, category LowCardinality(String), dwell_ms UInt32, viewed_at DateTime ) ENGINE = MergeTree() ORDER BY (user_id, domain, viewed_at);
Step 2 — AggMT target (stores partial states, not final numbers)
CREATE TABLE pv_user_profile ( user_id String, domain String, category LowCardinality(String), view_count AggregateFunction(count, UInt8), total_dwell_ms AggregateFunction(sum, UInt32), last_seen AggregateFunction(max, DateTime) ) ENGINE = AggregatingMergeTree() ORDER BY (user_id, domain, category);
Step 3 — MV: written once, fires on every INSERT forever
CREATE MATERIALIZED VIEW pv_mv TO pv_user_profile AS -- "TO" routes output to AggMT SELECT user_id, domain, category, countState() AS view_count, -- partial state, not a number sumState(dwell_ms) AS total_dwell_ms, maxState(viewed_at) AS last_seen FROM page_views GROUP BY user_id, domain, category; -- ☝ Only runs on the NEW rows from each INSERT (the delta)
Step 4 — Query: *Merge() finalises the accumulated states
SELECT domain, category, countMerge(view_count) AS total_views, round(sumMerge(total_dwell_ms) / 1000.0) AS avg_dwell_sec, maxMerge(last_seen) AS last_seen FROM pv_user_profile WHERE user_id = 'user_001' GROUP BY domain, category ORDER BY total_views DESC LIMIT 10;
✅ Delta-only processing
The MV SELECT runs only on the new rows from each INSERT. Insert 25 rows → MV processes 25 rows, not the full 50,000-row table.
✅ Partial states stack over time
Each INSERT appends new state rows to the AggMT. countMerge() finalises across all accumulated partial states at query time — background merges compact them silently.
⚠️ vs PostgreSQL REFRESH MATERIALIZED VIEW
PostgreSQL re-runs the full query on the full table on a schedule. ClickHouse fires on each INSERT — zero lag, no refresh, no stale window.
⚡ Real-time feeds 📊 Pre-aggregation 🧠 No-cron rollups 🔁 Incremental ETL
ReplacingMergeTree

ReplacingMergeTree

Keeps only the latest version of each row with the same ORDER BY key. Great for upsert-style workloads. Deduplication happens asynchronously on merge.

-- Used for: error_summary CREATE TABLE error_summary ( date Date, service LowCardinality(String), error_msg String, count UInt32, version UInt64 -- higher wins ) ENGINE = ReplacingMergeTree(version) ORDER BY (date, service, error_msg); -- "Upsert" by re-inserting with new version: INSERT INTO error_summary VALUES (today(), 'api', 'timeout', 42, now64());
⚠️ Deduplication 🔄 Upserts
CollapsingMergeTree

CollapsingMergeTree

Uses a sign column (+1/-1) to cancel previous rows. Allows efficient in-place corrections without mutations — perfect for billing adjustments.

-- Used for: budget_limits CREATE TABLE budget_limits ( date Date, team LowCardinality(String), budget_usd Float64, sign Int8 -- +1 insert, -1 cancel ) ENGINE = CollapsingMergeTree(sign) ORDER BY (date, team); -- Correct a budget: cancel then re-insert INSERT INTO budget_limits VALUES ('2024-01-01', 'core-platform', 5000, -1), ('2024-01-01', 'core-platform', 7500, +1);
🔁 Corrections 💳 Billing
ReplicatedMergeTree

ReplicatedMergeTree

All MergeTree variants have a Replicated counterpart. Data is synchronised across shards via ZooKeeper / ClickHouse Keeper for high availability.

-- Production HA setup on 2+ nodes: CREATE TABLE events ON CLUSTER my_cluster ( timestamp DateTime, service String ) ENGINE = ReplicatedMergeTree( '/clickhouse/tables/{shard}/events', '{replica}' -- auto-substituted ) ORDER BY (service, timestamp); -- Reads+writes auto-fan-out to all replicas

Sharding & Replication

Two fundamental ClickHouse cluster concepts — Sharding splits data horizontally across nodes for scale, while Replication duplicates data across nodes for high availability. This live demo runs a real 2-node cluster.

Sharding — Scale Out

Each shard owns a distinct subset of the data. A Distributed table fans writes out (using a sharding key hash) and merges reads from all shards transparently.

-- Sharding key assigns each row to a shard ENGINE = Distributed( demo_cluster, -- cluster name cluster_demo, -- database events_local, -- local table on each shard murmurHash3_32(user_id)-- sharding key ); -- user-42 always goes to shard 1 -- user-99 always goes to shard 2
📈 Horizontal Scale ⚡ Parallel Reads 🔀 Hash Routing

Replication — High Availability

Each replica holds the same data. ClickHouse Keeper (ZooKeeper-compatible) coordinates which replica is primary and syncs inserts automatically.

-- SAME ZK path on both replicas -- different {replica} macro per node ENGINE = ReplicatedMergeTree( '/clickhouse/tables/ha/events', -- shared ZK path '{replica}' -- replica-1 on node1 -- replica-2 on node2 ); -- Write to node1 → auto-synced to node2!
🔒 High Availability 🔄 Auto-Sync 💾 Keeper Coord.

Module 01

Why ClickHouse?

Understand what makes ClickHouse fundamentally different from traditional databases — and when to reach for it.

Columnar Storage OLAP vs OLTP Vectorized Execution
💡 Core Idea
Traditional databases store rows together on disk. ClickHouse stores columns together. When you run SELECT AVG(revenue) on a 100-column table, ClickHouse reads only the revenue column — not the other 99.

Row vs Columnar Storage

Row Store (Postgres, MySQL)
Disk: [id|name|revenue|ts][id|name|revenue|ts]... SELECT AVG(revenue) FROM events; → Must read ALL columns for every row → I/O: 100% of the table → Slow on analytics workloads
Columnar (ClickHouse)
Disk: [id][id]... [name][name]... [revenue][revenue]... SELECT AVG(revenue) FROM events; → Reads ONLY the revenue column → I/O: 1% of the table → 100x faster on the same hardware

Vectorized Execution

Instead of processing one row at a time, ClickHouse processes a batch of 8,192 values per CPU instruction using SIMD. Think of it as doing math on an entire column at once rather than looping through rows.

📊 When to Use ClickHouse
✅ Analytics queries (aggregations, GROUP BY)
✅ Time-series data (logs, metrics, events)
✅ Read-heavy workloads (dashboards, reports)
✅ Billions of rows with fast scans
❌ High-frequency point updates (use Postgres)
❌ Complex transactions (use Postgres/MySQL)
❌ Small data under 1M rows (any DB works)
❌ JOIN-heavy OLTP workloads

Compression — The Secret Weapon

Columnar storage makes compression extremely effective. A column of LowCardinality(String) with values like frontend, api-gateway, auth-service compresses to nearly nothing — because the same strings repeat millions of times.

-- ClickHouse typically achieves 5-20x compression -- Example: 100GB of raw logs → ~8GB on disk SELECT formatReadableSize(sum(bytes)) AS disk_size, formatReadableSize(sum(data_uncompressed_bytes)) AS raw_size, sum(data_uncompressed_bytes) / sum(bytes) AS ratio FROM system.parts WHERE database = 'demo' AND active;
Module 02

MergeTree: The Foundation

Every ClickHouse engine you'll use is built on MergeTree. Understanding how it stores and merges data is the single most important ClickHouse concept.

PartsMergesORDER BYSparse Index
💡 Core Idea
ClickHouse never updates rows in-place. Every INSERT creates a new "part" (a folder of column files). Background merge jobs then combine small parts into larger, sorted ones. This is why ClickHouse is fast at INSERTs AND queries — writes are always sequential appends.

The Parts Lifecycle

-- After 3 INSERTs you have 3 parts: /data/demo/telemetry_events/ 20240101_1_1_0/ ← part from INSERT #1 (sorted by ORDER BY) 20240101_2_2_0/ ← part from INSERT #2 20240101_3_3_0/ ← part from INSERT #3 -- ClickHouse background thread merges them: 20240101_1_3_1/ ← merged part (all 3 → 1, fully sorted) (old parts deleted after merge completes) -- Each part contains one file per column: timestamp.bin ← compressed timestamp values service.bin ← compressed service values primary.idx ← sparse primary index (1 entry per 8192 rows)
⚠️ The Most Important Schema Decision
ORDER BY in ClickHouse is NOT a primary key — it is the physical sort order on disk. This determines which queries run fast. A query that filters on your ORDER BY columns skips most of the data using the sparse index. A query that filters on other columns scans everything.

Choosing ORDER BY

Wrong ORDER BY
ORDER BY (id) -- random IDs → bad -- "Show errors for api-gateway" WHERE service = 'api-gateway' → Must scan ALL rows because service is not the sort key → Full table scan every time
Right ORDER BY
ORDER BY (service, timestamp) -- ✓ -- "Show errors for api-gateway" WHERE service = 'api-gateway' → Sparse index skips directly to api-gateway rows → Reads only relevant granules

PARTITION BY — Data Lifecycle

PARTITION BY toYYYYMM(timestamp) splits data into monthly folders. This lets you drop an entire month with ALTER TABLE DROP PARTITION '202401' — instant, no scanning needed.

CREATE TABLE app_logs ( timestamp DateTime, service LowCardinality(String), level LowCardinality(String), message String ) ENGINE = MergeTree() ORDER BY (service, level, timestamp) -- ← most-filtered columns first PARTITION BY toYYYYMM(timestamp) -- ← monthly partitions for lifecycle TTL timestamp + INTERVAL 90 DAY; -- ← auto-delete after 90 days
Module 03

Engine Variants

All ClickHouse engines inherit from MergeTree and differ only in what happens during background merges.

SummingMergeTreeAggregatingMergeTreeReplacingMergeTreeCollapsingMergeTree
💡 The Pattern
Every engine variant adds one behavior to the merge step: "when two parts are being merged and two rows share the same ORDER BY key, do X." The rest is identical to MergeTree.

SummingMergeTree — Auto-sum counters

During merge, rows with the same ORDER BY key are collapsed and their numeric columns are summed. Perfect for counters and accumulators that you'll always query with GROUP BY anyway.

-- cost_usage uses SummingMergeTree ENGINE = SummingMergeTree((cost_usd, api_calls, tokens_used)) ORDER BY (team, service, toDate(timestamp)) -- Three raw inserts: -- (platform, api-gateway, 0.05, 100) -- (platform, api-gateway, 0.03, 80) -- (platform, api-gateway, 0.08, 120) -- After merge: (platform, api-gateway, 0.16, 300) ← auto-summed! -- Query: always use sum() for correctness (parts may not all be merged yet) SELECT service, sum(cost_usd) AS total_cost FROM demo.cost_usage GROUP BY service ORDER BY total_cost DESC;

ReplacingMergeTree — Upserts / Deduplication

During merge, rows with the same ORDER BY key are collapsed to the latest version. The version column (usually a timestamp or auto-increment) determines which row wins.

-- error_summary uses ReplacingMergeTree ENGINE = ReplacingMergeTree(version) -- higher version wins ORDER BY (date, service, error_msg) -- "Upsert" pattern: re-insert with a higher version number INSERT INTO error_summary VALUES (today(), 'api-gateway', 'timeout', 42, toUnixTimestamp(now())); -- After merge, only the latest version survives -- Note: FINAL forces dedup at query time (slower) if not yet merged SELECT * FROM error_summary FINAL WHERE service = 'api-gateway';

CollapsingMergeTree — Corrections with sign

Uses a sign column (+1 or -1). During merge, +1 rows cancel out -1 rows with the same ORDER BY key. This allows you to "delete" or "correct" a value by inserting a -1 row — no mutations needed.

-- budget_limits uses CollapsingMergeTree ENGINE = CollapsingMergeTree(sign) ORDER BY (date, team) -- Original budget: INSERT INTO budget_limits VALUES ('2024-01-01', 'platform', 5000.0, +1); -- Correct it (cancel old, insert new): INSERT INTO budget_limits VALUES ('2024-01-01', 'platform', 5000.0, -1), -- cancel old ('2024-01-01', 'platform', 7500.0, +1); -- new value -- Query the NET budget: SELECT team, sum(budget_usd * sign) AS effective_budget FROM budget_limits GROUP BY team HAVING effective_budget > 0;
Module 04

Materialized Views & Approximations

Pre-compute results at insert time so dashboards query in milliseconds — plus learn how ClickHouse estimates COUNT DISTINCT in constant memory.

Materialized ViewsHyperLogLog-State/-Merge Combinators
💡 Core Idea
A Materialized View (MV) in ClickHouse is like a trigger — every INSERT into the source table automatically runs a SELECT and inserts the result into a target table. The target table is usually an AggregatingMergeTree that stores partial aggregate states rather than final values.

Materialized View + AggregatingMergeTree

-- Step 1: target table stores partial states (not final numbers) CREATE TABLE telemetry_hourly_agg ( hour DateTime, service LowCardinality(String), event_count AggregateFunction(count), -- compressed partial HLL state unique_users AggregateFunction(uniq, String) -- HyperLogLog sketch ) ENGINE = AggregatingMergeTree() ORDER BY (service, hour); -- Step 2: MV fires on every INSERT into telemetry_events CREATE MATERIALIZED VIEW mv_telemetry_hourly TO telemetry_hourly_agg AS SELECT toStartOfHour(timestamp) AS hour, service, countState() AS event_count, -- ← -State suffix! uniqState(user_id) AS unique_users FROM telemetry_events GROUP BY hour, service; -- Step 3: query uses -Merge suffix to combine partial states SELECT hour, service, countMerge(event_count) AS events, uniqMerge(unique_users) AS distinct_users FROM telemetry_hourly_agg GROUP BY hour, service ORDER BY hour DESC;
📐 The -State / -Merge Pattern
Any aggregate function in ClickHouse has two combinators:
  • uniqState() — stores a HyperLogLog sketch (compact binary blob) instead of the final count
  • uniqMerge() — combines multiple sketches and returns the final estimated count
  • This lets you pre-aggregate at insert time and merge at query time

HyperLogLog: Counting Billions in Bytes

uniq() uses HyperLogLog to count distinct values in constant memory (~2KB per group), regardless of cardinality. Great for "unique users" at web scale. Need exact? Use uniqExact() — but it's 100x more memory.

-- HLL approximation (fast, O(1) memory) SELECT service, uniq(user_id) AS approx_users FROM telemetry_events GROUP BY service; -- Exact count (slow, O(n) memory) SELECT service, uniqExact(user_id) AS exact_users FROM telemetry_events GROUP BY service; -- Practical rule: <1% error from HLL is acceptable for analytics -- Use uniqExact() only for billing/compliance where exactness matters
Module 05

Structured Logging & TTL

ClickHouse is one of the best log stores available — auto-expiring hot logs, instant aggregations, and sub-second text search.

TTLLowCardinalityLIKE SearchLog Lifecycle
💡 Core Idea
TTL (Time-To-Live) is a built-in ClickHouse feature that automatically deletes (or moves to cold storage) old rows based on a time expression. No cron jobs, no manual cleanup — it runs as a background operation during merges.

TTL Expression

CREATE TABLE app_logs ( timestamp DateTime, level LowCardinality(String), -- ← saves 50-90% space vs String service LowCardinality(String), message String ) ENGINE = MergeTree() ORDER BY (service, level, timestamp) PARTITION BY toDate(timestamp) TTL timestamp + INTERVAL 90 DAY -- ← delete rows older than 90 days DELETE -- (default behaviour) SETTINGS merge_with_ttl_timeout = 3600; -- check every hour -- You can also MOVE to cheaper storage tiers: TTL timestamp + INTERVAL 30 DAY TO DISK 'ssd', timestamp + INTERVAL 90 DAY TO DISK 's3';

LowCardinality — Automatic Dictionary Encoding

When a String column has fewer than ~10,000 unique values (like log levels or service names), wrap it in LowCardinality(). ClickHouse automatically stores it as an integer dictionary — often 10x less space and 2x faster queries.

-- Without LowCardinality: stores full string for every row level String -- 'ERROR', 'ERROR', 'ERROR'... stored 1M times -- With LowCardinality: dictionary encoding level LowCardinality(String) -- Dictionary: {0: 'DEBUG', 1: 'INFO', 2: 'WARN', 3: 'ERROR'} -- Column: [3, 3, 1, 3, 2, 1, ...] ← just integers! -- Works transparently — no query changes needed: WHERE level = 'ERROR' -- still works exactly the same

Log Search Patterns

-- Fast: prefix match on the ORDER BY key SELECT * FROM app_logs WHERE service = 'api-gateway' AND level = 'ERROR' ORDER BY timestamp DESC LIMIT 100; -- LIKE search (full-text, slightly slower — scans column values) SELECT * FROM app_logs WHERE message LIKE '%timeout%' AND timestamp >= now() - INTERVAL 1 HOUR; -- Error rate by service (instant due to ORDER BY) SELECT service, countIf(level = 'ERROR') AS errors, count() AS total, round(100.0 * countIf(level='ERROR') / count(), 2) AS error_pct FROM app_logs WHERE timestamp >= now() - INTERVAL 24 HOUR GROUP BY service ORDER BY error_pct DESC;
Module 06

Cost Tracking & Aggregations

Build a self-maintaining cost dashboard: SummingMergeTree auto-aggregates spend, while CollapsingMergeTree handles budget corrections without mutations.

SummingMergeTreeCollapsingMergeTreeBudget Alerts
💡 Core Idea
SummingMergeTree is a zero-cost aggregation layer. Every time ClickHouse merges parts, it automatically sums your numeric columns for rows with the same ORDER BY key. No ETL job, no cron tab, no scheduled aggregation — the DB maintains the totals itself.

SummingMergeTree: How Totals Stay Current

-- cost_usage: appended continuously by services ENGINE = SummingMergeTree((cost_usd, api_calls, tokens_used)) ORDER BY (team, service, toDate(timestamp)) -- 100 inserts happen during the day: INSERT INTO cost_usage VALUES ('2024-01-15', 'api-gateway', 'platform', 0.0012, 12, 800); INSERT INTO cost_usage VALUES ('2024-01-15', 'api-gateway', 'platform', 0.0008, 8, 500); -- ... 98 more inserts ... -- Background merge collapses them automatically: -- Result: ('2024-01-15', 'api-gateway', 'platform', SUM(cost), SUM(calls), SUM(tokens)) -- Always query with sum() for correctness (some parts may not be merged yet): SELECT team, service, sum(cost_usd) AS total_cost FROM cost_usage GROUP BY team, service ORDER BY total_cost DESC;

Budget Alerts with CollapsingMergeTree

-- Check: which teams exceeded their daily budget? SELECT cu.service, cu.team, sum(cu.cost_usd) AS daily_spend, -- CollapsingMergeTree net budget (sign=+1 adds, sign=-1 cancels) sum(bl.budget_usd * bl.sign) AS budget, round(100.0 * sum(cu.cost_usd) / sum(bl.budget_usd * bl.sign), 1) AS used_pct FROM cost_usage cu JOIN budget_limits bl ON cu.team = bl.team AND bl.date = toDate(cu.timestamp) WHERE cu.timestamp >= today() AND bl.sign = 1 GROUP BY cu.service, cu.team HAVING daily_spend > budget * 0.8 -- alert at 80% ORDER BY used_pct DESC;
Module 07

Cluster & Replication

Scale out horizontally with sharding, and achieve high availability with replication — understand the difference and when you need each.

Distributed TableShardingReplicatedMergeTreeClickHouse Keeper
💡 Two Different Concepts
Sharding = split different data across nodes (for scale). Replication = copy the same data across nodes (for availability). Production clusters usually use both — 2 shards × 2 replicas = 4 nodes total.

Sharding: The Distributed Table

A Distributed table is a virtual table — it contains no data itself. INSERTs fan out to local tables on each shard; SELECTs fan out and merge. The sharding key (usually a hash of a user ID) determines which shard each row goes to.

-- Local table on EACH shard node: CREATE TABLE events_local ON CLUSTER demo_cluster (...) ENGINE = ReplicatedMergeTree('/ch/tables/{shard}/events', '{replica}') ORDER BY (user_id, timestamp); -- Distributed table: virtual, one per cluster CREATE TABLE events_dist ON CLUSTER demo_cluster (...) ENGINE = Distributed( demo_cluster, -- cluster name from config.xml default, -- database events_local, -- local table on each shard murmurHash3_32(user_id) -- sharding key → deterministic routing ); -- Write to distributed → auto-routes to correct shard: INSERT INTO events_dist VALUES ('user-42', ...); -- murmurHash3_32('user-42') % 2 = shard 1 → goes to clickhouse-node1 -- Read from distributed → fan-out, parallel scan, merge: SELECT count(), service FROM events_dist GROUP BY service;

Replication: ClickHouse Keeper

ReplicatedMergeTree uses ClickHouse Keeper (a ZooKeeper-compatible service) to track a shared INSERT log. When node1 inserts data, Keeper records the log entry — node2 reads it and applies the same insert. This is asynchronous and happens within milliseconds.

-- Same ZK path, different replica macro → two replicas of same data CREATE TABLE events_replicated ( timestamp DateTime, service String, user_id String ) ENGINE = ReplicatedMergeTree( '/clickhouse/tables/ha/events_replicated', -- shared ZK path '{replica}' -- expands to 'replica-1' on node1, 'replica-2' on node2 ) ORDER BY (service, timestamp); -- How Keeper coordinates it: -- 1. INSERT arrives at replica-1 (node1) -- 2. node1 writes part locally AND logs entry to Keeper -- 3. node2 reads Keeper log → fetches the part from node1 -- 4. Both replicas now have identical data -- Check replication status: SELECT replica_name, is_leader, queue_size, inserts_in_queue FROM system.replicas WHERE table = 'events_replicated';

13 Common ClickHouse Mistakes

A curated breakdown of the most common pitfalls new ClickHouse users encounter — what causes them, what goes wrong, and exactly how to fix them. Based on the official ClickHouse engineering blog.

Filter by category:
01
Too Many Parts
🚢 Ingestion
🔴 DB::Exception: Too many parts (N in total)
❌ The Mistake

Three common causes: (1) High-cardinality partition key — using date_time_ms as a partition key creates thousands of folders; parts in different partitions are never merged. (2) Many tiny inserts — every single INSERT creates a new part. (3) Too many materialized views — each MV adds part pressure on inserts.

⚠️ What Goes Wrong

Queries slow down because more parts = more index files to evaluate. In extreme cases ClickHouse throws Too Many Parts and rejects further inserts. High part counts also cause slower startup and increased Keeper load in replicated setups.

✅ The Fix
❌ Wrong — high-cardinality partition
CREATE TABLE events ( ts DateTime, service String ) ENGINE = MergeTree() -- BAD: millions of unique values! PARTITION BY toStartOfMinute(ts) ORDER BY (service, ts);
✅ Right — low-cardinality partition + batched inserts
CREATE TABLE events ( ts DateTime, service String ) ENGINE = MergeTree() -- GOOD: < 1000 unique partition values PARTITION BY toYYYYMM(ts) ORDER BY (service, ts); -- Batch inserts: 10k–100k rows at a time -- OR use async inserts: INSERT INTO events SETTINGS async_insert = 1, wait_for_async_insert = 1 VALUES (...);
🔬 Try it live — 15 individual INSERTs vs 1 batch INSERT
Click ▶ Run ❌ Wrong to see the mistake
Click ▶ Run ✅ Fixed to see the solution
02
Going Horizontal Too Early
⚙️ Ops
❌ The Mistake

New users immediately deploy dozens of ClickHouse nodes on Kubernetes, expecting horizontal scaling to be the answer — just like with stateless services.

⚠️ What Goes Wrong

Unnecessary operational complexity, higher costs, and potentially worse query performance due to JOIN data being shuffled over the network. ClickHouse is designed to saturate a single large machine (hundreds of cores, TBs of RAM).

✅ The Fix
💡 Scale Vertically First
ClickHouse parallelizes every query stage across all CPU cores by default (max_threads = number of cores). A single node with 64 cores and 512 GB RAM will comfortably handle billions of rows. Only add nodes for redundancy (2 replicas) or when a single machine is genuinely exhausted.
03
Mutation Pain
⚙️ Ops
❌ The Mistake

Running frequent ALTER TABLE ... UPDATE/DELETE (classic mutations) to modify data — treating ClickHouse like an OLTP database where updates are routine.

⚠️ What Goes Wrong

Classic mutations rewrite entire data parts. They compete with background merges for CPU/IO. On replicated clusters, every replica applies the mutation independently. This cascades into the "Too Many Parts" problem, replication delays, and extremely high resource usage.

✅ The Fix
❌ Slow — classic mutation rewrites full parts
-- Rewrites ALL parts containing this service! ALTER TABLE events UPDATE status = 'archived' WHERE service = 'legacy-api';
✅ Fast — lightweight update (patch parts)
-- Writes only a compact "patch part" for -- the changed column values. Up to 1000x faster! UPDATE events SET status = 'archived' WHERE service = 'legacy-api'; -- For deletes, use lightweight DELETE: DELETE FROM events WHERE ts < now() - INTERVAL 90 DAY;
04
Mishandling Semi-Structured Data
🗂 Schema
❌ The Mistake

Storing JSON as a raw String column and parsing it at query time with JSONExtractString(), or blindly using Map(String, String) for everything. Both sacrifice compression and query speed.

⚠️ What Goes Wrong

String columns can't be compressed efficiently by ClickHouse's columnar codec. Parsing JSON at query time is slow and error-prone. Map columns store data as arrays, which is harder to index.s

✅ The Fix
❌ Slow — raw JSON string parsed at query time
CREATE TABLE logs ( ts DateTime, -- ❌ no compression, slow extraction raw_json String ) ENGINE = MergeTree() ORDER BY ts; -- Query must parse on every read: SELECT JSONExtractString(raw_json, 'user_id') FROM logs WHERE ts > now() - INTERVAL 1 DAY;
✅ Best — explicit columns (known schema)
CREATE TABLE logs ( ts DateTime, -- ✅ typed, compressed, indexed user_id String, level LowCardinality(String), message String ) ENGINE = MergeTree() ORDER BY (level, ts); -- OR for truly dynamic schemas, use JSON type: CREATE TABLE logs ( ts DateTime, payload JSON -- native JSON type ) ENGINE = MergeTree() ORDER BY ts;
05
Overuse of Nullable
🗂 Schema
❌ The Mistake

Defaulting to Nullable(String), Nullable(UInt64), etc. for any column that might be empty — just like in a traditional SQL database.

⚠️ What Goes Wrong

Every Nullable column creates a hidden secondary UInt8 "null mask" column that must be read alongside the data column on every query. This doubles storage I/O for those columns and almost always hurts performance.

✅ The Fix
❌ Nullable everywhere — extra hidden null-mask column
CREATE TABLE users ( id UInt64, email Nullable(String), -- ❌ extra null column age Nullable(UInt8), -- ❌ extra null column country Nullable(String) -- ❌ extra null column ) ENGINE = MergeTree() ORDER BY id;
✅ Use sentinel default values instead
CREATE TABLE users ( id UInt64, email String DEFAULT '', -- ✅ empty string = "null" age UInt8 DEFAULT 0, -- ✅ 0 = "unknown" country LowCardinality(String) DEFAULT '' ) ENGINE = MergeTree() ORDER BY id; -- Query: filter out sentinel values SELECT * FROM users WHERE email != '';
🔬 Try it live — Nullable(String) vs String DEFAULT '' storage cost
Click ▶ Run ❌ Wrong to create a Nullable table with 2k rows
Click ▶ Run ✅ Fixed to compare with DEFAULT values
06
Insert-Time Deduplication Surprise
🚢 Ingestion
😮 Inserted 8 rows twice … but SELECT only shows 4 rows!
❌ The Mistake

On replicated tables (including ClickHouse Cloud), ClickHouse hashes each INSERT block. If the exact same block is submitted again (same rows, same order, same size), it is silently ignored. Users are surprised when retried inserts "disappear".

⚠️ What Goes Wrong

Not a bug — it's intentional idempotency for safe insert retries. But if you expect duplicate rows (e.g., reprocessing events), this will silently drop them. The deduplication window is the last 100 blocks by default (replicated_deduplication_window = 100).

✅ The Fix
-- Understand the behavior: -- Block hash is stored in Keeper for last N blocks. -- Identical insert = same hash → silently deduplicated. -- To ALLOW duplicate data, change block sizes or -- add a random nonce column so hashes differ: INSERT INTO events SELECT *, rand() as nonce FROM ...; -- To DISABLE deduplication on non-replicated tables: CREATE TABLE events (...) ENGINE = MergeTree() ORDER BY ts SETTINGS non_replicated_deduplication_window = 0; -- Or check: how many blocks are being tracked? SELECT * FROM system.zookeeper WHERE path LIKE '/clickhouse/tables/%/blocks';
🔬 Try it live — Retry inserts on non-replicated vs ReplacingMergeTree
Click ▶ Run ❌ Wrong — inserts same block twice, counts the result
Click ▶ Run ✅ Fixed — shows ReplacingMergeTree + SELECT FINAL
07
Poor Primary Key Selection
🗂 Schema
❌ The Mistake

Choosing a primary key based on OLTP instincts — putting a high-cardinality UUID first, or picking columns that don't match actual query filters. ClickHouse's sparse index is fundamentally different from a B-tree.

⚠️ What Goes Wrong

A poorly ordered primary key means ClickHouse cannot skip granules efficiently → full table scans even for simple queries. It also hurts compression because ClickHouse compresses better when data is sorted by low-cardinality columns first.

✅ The Fix
❌ UUID first — high cardinality, kills compression
CREATE TABLE events ( id UUID, service LowCardinality(String), ts DateTime, level LowCardinality(String) ) ENGINE = MergeTree() -- ❌ High-cardinality UUID first ORDER BY (id, service, ts);
✅ Low → high cardinality order = best compression + skipping
CREATE TABLE events ( id UUID, service LowCardinality(String), -- ~10 values level LowCardinality(String), -- ~5 values ts DateTime, -- millions of values user_id String -- billions of values ) ENGINE = MergeTree() -- ✅ Low cardinality first → best compression -- ✅ Match your WHERE clause columns ORDER BY (service, level, ts); -- Rule: put columns you filter on most in ORDER BY. -- Order by ascending cardinality within those.
🔬 Try it live — EXPLAIN: WHERE user_id vs WHERE service (PK column)
Filter on user_id — not in primary key → full scan
Filter on service — first ORDER BY column → index skip
08
Overuse of Data Skipping Indices
🔍 Query
❌ The Mistake

Adding bloom filter or minmax skip indices to every column that appears in a WHERE clause, hoping it will speed up all queries automatically.

⚠️ What Goes Wrong

Skip indices only help when there's a strong correlation between the primary key column and the indexed column. Without correlation, the index matches most granules anyway (reads everything) — incurring index overhead with zero benefit. They also slow INSERT performance because the index must be updated.

✅ The Fix
⚠️ Order of operations for query optimization
1st: Improve PRIMARY KEY / ORDER BY to match query patterns.
2nd: Add a Materialized View with a different ORDER BY for alternate access patterns.
3rd: Consider Projections for pre-aggregated data.
Last resort only: Data skipping indices — only when there's high correlation between primary key and indexed column.
09
LIMIT Doesn't Always Short-Circuit
🔍 Query
❌ The Mistake

Assuming LIMIT 10 always makes queries faster — applying OLTP intuition where fewer results = faster execution.

⚠️ What Goes Wrong

For aggregations, ClickHouse must scan the entire table before it can apply LIMIT. A query like SELECT a, count() FROM t GROUP BY a ORDER BY b LIMIT 1 scans 27 million rows even though you only want 1 row back.

✅ The Fix
❌ Full scan — 3 sec, 27M rows read
SELECT postcode1, postcode2, avg(price) AS avg_price FROM uk_price_paid GROUP BY postcode1, postcode2 LIMIT 1; -- Elapsed: 3.028 sec, read 27.55M rows
✅ Early termination — 1 sec, 4.8M rows read
SELECT postcode1, postcode2, avg(price) AS avg_price FROM uk_price_paid GROUP BY postcode1, postcode2 LIMIT 1 -- Enable in-order aggregation short-circuit: SETTINGS optimize_aggregation_in_order = 1; -- Elapsed: 0.999 sec, read 4.81M rows (83% less!)
🔬 Try it live — GROUP BY + LIMIT 1: timing with and without early termination
Default — full aggregation before LIMIT
optimize_aggregation_in_order=1 — early termination
10
Readonly Tables
⚙️ Ops
🔴 DB::Exception: Table is in readonly mode
❌ The Mistake

In self-managed replicated setups, co-locating ClickHouse Keeper (the coordination service) on the same machine as ClickHouse, or under-allocating Keeper memory/CPU.

⚠️ What Goes Wrong

When a node loses connection to Keeper (coordination service), it enters read-only mode — all INSERT/ALTER statements fail. This happens most often when Keeper is resource-starved or unreachable. Non-Cloud deployments must manage this carefully.

✅ The Fix
💡 Keeper best practices
• Run ClickHouse Keeper on dedicated hardware — not mixed with ClickHouse itself.
• Deploy Keeper in a 3-node quorum for high availability.
• Prefer ClickHouse Keeper over ZooKeeper — it's purpose-built, written in C++ (no JVM heap tuning).
• ClickHouse Cloud users: this is fully managed — not your problem! ☁️
11
Memory Limit Exceeded
🔍 Query
🔴 DB::Exception: Memory limit (total) exceeded: would use X, maximum: Y
❌ The Mistake

Running large JOINs that put the right-hand table (build side) in memory, or doing high-cardinality GROUP BY aggregations that build massive in-memory hash maps. Also: leaving rogue queries running without resource limits.

⚠️ What Goes Wrong

ClickHouse kills the query to protect the server. In severe cases, multiple concurrent heavy queries can OOM the server itself. Common triggers: GROUP BY on UUID or user_id (billions of unique keys), large JOINs not using the small-table-right rule.

✅ The Fix
-- 1. Put smaller table on the RIGHT side of JOIN: SELECT * FROM large_events e JOIN small_users u ON e.user_id = u.id; -- ✅ small on right -- 2. Enable disk-spill for aggregations: SET max_bytes_before_external_group_by = 10000000000; -- 10GB -- 3. Enable disk-spill for JOINs: SET max_bytes_before_external_sort = 10000000000; -- 4. Kill rogue queries by user: SELECT query_id, user, elapsed, memory_usage FROM system.processes ORDER BY memory_usage DESC; KILL QUERY WHERE user = 'data_team' AND elapsed > 60; -- 5. Set per-user memory quotas in users.xml or via SQL: ALTER USER analyst SETTINGS max_memory_usage = 5000000000;
🔬 Try it live — Heavy aggregation: no safety nets vs disk-spill settings
No memory limits — could OOM on large tables
max_bytes_before_external_group_by — safe disk spill
12
Materialized View Pitfalls
📐 Mat. Views
❌ The Mistakes (4 common ones)

a) Thinking MVs see the full table — they only fire on new INSERT blocks.
b) Attaching 50+ MVs to a single table — each adds insert overhead + part pressure.
c) Using CPU-heavy state functions (e.g., quantileState) on every insert.
d) Mismatched schema between the MV SELECT and the target AggregatingMergeTree table.

⚠️ What Goes Wrong

Slow inserts, cascading "Too Many Parts" from all the MV target tables, silent data corruption when schemas don't match, or MVs that don't reflect mutations/partition drops on the source table.

✅ The Fix
❌ Schema mismatch — column order relied on, not names
-- Target table: CREATE TABLE summary ( day Date, service String, visits AggregateFunction(sum, Int64) ) ENGINE = AggregatingMergeTree() ORDER BY (day, service); -- ❌ MV column order doesn't match target column names! CREATE MATERIALIZED VIEW mv_summary TO summary AS SELECT toDate(ts) AS day, service, sumState(1) AS cnt -- "cnt" != "visits"! FROM events GROUP BY day, service;
✅ Explicit aliases matching target table exactly
CREATE MATERIALIZED VIEW mv_summary TO summary AS SELECT toDate(ts) AS day, service, -- ✅ alias matches target column name exactly sumState(1) AS visits FROM events GROUP BY day, service; -- ✅ Enable parallel view processing to reduce latency: SET parallel_view_processing = 1;
🔬 Try it live — MV created after data (0 rows) vs backfill via INSERT INTO SELECT
Create MV after existing data — see how many rows it captured
Run the backfill INSERT INTO ... SELECT to populate historical data
13
Experimental Features in Production
⚙️ Ops
❌ The Mistake

Building core production functionality on top of features marked experimental (not even beta) — such as SET allow_experimental_variant_type = 1 or early JSON type builds.

⚠️ What Goes Wrong

Experimental features can change APIs without notice, be deprecated, or be removed entirely in a future release. On-disk formats may not be forward-compatible, meaning a ClickHouse upgrade could corrupt or make your tables unreadable. They are officially unsupported by the ClickHouse team.

✅ The Fix
⚠️ Feature Maturity Levels
🟢 Production-ready: No special setting required. Fully supported, stable API.
🟡 Beta: SET allow_beta_... — On a path to production, officially supported but may have edge cases.
🔴 Experimental: SET allow_experimental_... — Community prototypes. APIs can change. Do NOT build production systems on these.

Always check the ClickHouse docs feature maturity page before adopting new features.
